## TCP/IP 分层模型和 OSI 分层模型分别是怎样的？有什么区别

**OSI 七层模型**是理论标准，分得很细：物理、链路、网络、传输、会话、表示、应用。

**TCP/IP 模型**是根据互联网实际运行总结的四层：网络接口层、网际层、传输层、应用层。

**区别在于 TCP/IP：**

- 把底层两层合并成 **网络接口层**，不关注底层具体的传输
- 把上面三层合并成 **应用层**，不关注上层具体的会话管理

**原因：** OSI 太复杂，工程落地难，而 TCP/IP 更简洁、实践先行，所以互联网最终采用了 TCP/IP 模型。

> OSI 更适合学习，TCP/IP 更适合实际。

### 1. OSI 七层模型（理论标准）

| 层级 | 说明 |
|------|------|
| 物理层 | 硬件传输（网线、光纤、电信号）。 |
| 数据链路层 | 局域网内传输，靠 MAC 地址。 |
| 网络层 | 跨网寻址，靠 IP 地址。 |
| 传输层 | 端到端通信可靠性，TCP/UDP 在此。靠端口。 |
| 会话层 | 管理通信会话的建立与断开。 |
| 表示层 | 处理数据格式、加密、压缩。 |
| 应用层 | 直接面向用户的软件服务。 |

> 👉 特点：设计非常细致，但过于复杂，工程落地难。

### 2. TCP/IP 四层模型（实际运行）

| 层级 | 说明 |
|------|------|
| 网络接口层 | 物理层 + 数据链路层合并。TCP/IP 不关心底层硬件，只要能连上即可。 |
| 网际层（IP 层） | 负责数据包的路由和寻址。 |
| 传输层 | 端到端通信，TCP/UDP。通过端口号 |
| 应用层 | 会话层 + 表示层 + 应用层合并，交给应用程序自己处理。 |

> 👉 特点：更简洁，直接面向实际工程，互联网就是基于它运行。

## 什么是 HTTP 协议

HTTP 全称是超文本传输协议，是一种应用层的请求-响应协议，
本质上是 浏览器和服务器之间的沟通语言。

最初的 HTTP 它基于 TCP，因此传输可靠。（3.0改了）

**HTTP 有三个核心特点：请求响应；无状态；灵活**

第一，请求-响应模式，客户端主动、服务器被动；
第二，无状态，服务器不会记住客户端，需要 Cookie 和 Session 来维持会话；
第三，灵活性，能传输各种数据类型。

它的发展经历了 1.0 的简单连接、1.1 的长连接、2.0 的多路复用，到 3.0 的基于 UDP 的 QUIC 协议。

## HTTP 状态码有哪些

HTTP 状态码是服务器返回给客户端的处理结果报告。

**它分为五类：1xx 信息型，2xx 成功，3xx 重定向，4xx 客户端错误，5xx 服务器错误。**

**常见的有：** 200 成功，301 永久重定向，302 临时重定向，401 未认证，403 无权限，404 找不到资源，500 服务器错误。

| 分类 | 说明 |
|------|------|
| **1xx 信息型** | 请求已收到一部分，还在处理中（开发中很少遇到）。 |
| **2xx 成功** | 请求成功，最常见是 200 OK；206 Partial Content 用于断点续传。 |
| **3xx 重定向** | 资源搬家了，301 永久重定向（浏览器会缓存），302 临时重定向。 |
| **4xx 客户端错误** | 客户端请求有问题。 |
| **5xx 服务器错误** | 服务端的问题。 |

**4xx 客户端错误：**

- `400 Bad Request`：参数错误。
- `401 Unauthorized`：未认证/未登录。
- `403 Forbidden`：已认证但无权限。
- `404 Not Found`：资源不存在。

**5xx 服务器错误：**

- `500 Internal Server Error`：代码报错。
- `502 Bad Gateway`：网关错误，上游服务没响应

## 对比 GET 和 POST 请求方式

GET 和 POST 的区别首先是是参数放哪儿。

|  | 说明 |
|---|------|
| **GET** | 参数放在 URL 中，像明信片，内容直接暴露。缺点是 URL 有长度限制（约 2KB），不适合传敏感信息。 |
| **POST** | 参数放在请求体里，像快递盒子封起来。没有大小限制，支持多种数据格式。 |

**更重要的是语义、幂等性和缓存行为。**

**语义上的区别：**

- **GET：** 语义是只读，用来获取资源，不应该修改服务器数据。
- **POST：** 语义是提交数据，让服务器处理，比如写入数据库，会改变服务器状态。

**幂等性和缓存行为：**

- **GET：** 幂等，多次请求结果一样，不会改变服务器数据。浏览器会缓存 GET 请求结果，因为它是只读的。
- **POST：** 非幂等，多次请求可能产生重复数据（比如重复订单）。默认不会缓存，因为有副作用，每次结果可能不同。

> 这三点才是完整的回答 GET 和 POST 的区别。

## Http 与 Https 的区别？为什么 Http 是可靠的协议？

**整体概括：** HTTP 是明文传输，HTTPS 在 HTTP 基础上加了加密和认证。

**具体区别：**

| 对比项 | HTTP | HTTPS |
|--------|------|-------|
| 端口 | 默认端口是 80 | 默认端口是 443 |
| 传输安全性 | 明文，容易被窃听和篡改 | 通过 SSL/TLS 协议加密，保证数据安全 |
| 加密机制 | 无 | 握手阶段用非对称加密先来交换会话密钥，后续数据传输用对称加密保证效率，同时通过 CA 证书验证服务器身份 |

**客户端的作用：**

- 客户端生成一个随机的会话密钥（后续用来做对称加密）
- 用服务器的公钥加密后发给服务器

**服务器的作用：**

- 用自己的私钥解密，得到会话密钥
- 至此，客户端和服务器都持有同一个会话密钥

> 严格来说，现在用的都是 TLS，SSL 已经被淘汰。

## 浏览器输入 URL 到显示页面的完整过程？简述一次 http 网络请求的过程？

**整体概括：**
整个过程可以概括为四步：找地址 → 建连接 → 拿数据 → 画页面。

### 1. 先做 DNS 解析找到服务器地址

- 浏览器输入域名，机器只认 IP，需要先解析。
- **查找顺序：** 浏览器缓存 → 操作系统缓存/hosts 文件 → 本地 DNS →
  没有的话去访问，根域名服务器 → 顶级域名服务器 → 权威域名服务器。
- 最终得到目标服务器的 IP 地址。

### 2. 通过 TCP 三次握手建立连接，如果是 HTTPS 还要做 TLS 加密

- 有了 IP，客户端和服务器建立 TCP 连接（三次握手）。
- 如果是 HTTPS，还要进行 TLS 握手：协商加密算法、交换密钥，确保后续通信加密。现在一般都是https

### 3. 浏览器发送 HTTP 请求，服务器返回响应

- 浏览器构建 HTTP 请求报文，带上请求头、Cookie 等信息。
- 服务器处理请求，查数据库、读文件，返回 HTTP 响应报文（通常是 HTML）。

### 4. 浏览器解析 HTML 和 CSS，完成布局和绘制，把页面展示出来

```
解析 HTML   → 构建 DOM 树。
解析 CSS    → 构建 CSSOM 树。
合并        → 渲染树（display:none 元素不会进入渲染树）。
布局（回流） → 计算元素位置和大小。
绘制        → 填充颜色、文字、图片，最终显示页面。
```

## DNS 的查询过程是怎样的

DNS 查询过程分两阶段：

1. 先在本地找缓存，浏览器缓存 → 操作系统缓存/hosts 文件；找不到再往外找
2. 再交给服务器去查找，先由本地 DNS 服务器。
3. 本地 DNS 再去问根服务器、顶级域名服务器、权威服务器。最终权威服务器返回 IP，本地 DNS 缓存后再交给客户端

## DNS 是基于 TCP 还是 UDP

DNS 同时使用 UDP 和 TCP，但用在不同场景。

| 场景 | 协议 |
|------|------|
| 日常查询 | **UDP**（53 端口） |
| 大数据量同步或需要可靠传输 | **TCP**（53 端口） |

**为什么 DNS 常规查询用 UDP？**

- 🏎️ **追求速度：** DNS 查询是用户访问网页的第一步，越快越好。
- 🚫 **UDP 无需握手：** 没有三次握手，开销小。
- 📦 **UDP 头部只有 8 字节：** 报文短、效率高。
- 🔄 **一次请求一次响应：** 典型 DNS 查询只需要一个来回。

> 👉 所以 浏览器平时查域名走的是 UDP。

### UDP 的限制：512 字节

传统 DNS 协议规定：

- UDP 响应不能超过 512 字节
- 超过会被截断（TC=1）

这就引出了需要 TCP 的场景。

**必须用 TCP 的场景：**

| # | 场景 | 说明 |
|---|------|------|
| 1 | **区域传送（Zone Transfer）** | 主 DNS 与从 DNS 之间同步数据。一个区域可能有成百上千条记录，必须保证 不丢、不重、不乱序。所以必须用 TCP 的可靠传输。 |
| 2 | **响应包太大（超过 512 字节）** | 如果某次查询结果太大（例如 DNSSEC、TXT 记录很多），UDP 装不下。服务器会在 UDP 响应里设置 TC=1（Truncated），表示被截断。客户端看到 TC=1，会 自动重新发起 TCP 查询，拉取完整数据。 |

## TCP 和 UDP 的区别？

### 1. 连接机制

TCP 面向连接更严谨。UDP 无连接。

- **TCP：** 面向连接，必须先"三次握手"建立连接，传完数据还要"四次挥手"断开，过程严谨。
- **UDP：** 无连接，想发就发，不确认对方是否在线，像广播一样。

### 2. 可靠性

TCP 保证数据不丢、不重、不乱。UDP 可能存在一定的丢包乱序。

- **TCP：** 可靠传输，有 ACK 确认机制、超时重传、流量控制、拥塞控制。
- **UDP：** 不可靠传输，尽力而为，可能丢包、乱序，不做保证。

### 3. 传输方式

- **TCP：** 字节流模式，数据连续无边界，可能出现"粘包/拆包"，需要应用层自己处理边界。
- **UDP：** 数据包模式，每个包独立，有清晰边界，不会粘连。

### 4. 效率与开销

- **TCP：** 首部至少 20 字节，要维护连接状态，握手、挥手、重传逻辑，开销大，速度慢。
- **UDP：** 首部仅 8 字节，轻量快速，支持广播和组播，效率高。

### 5. 适用场景

- **TCP：** 适合对准确性要求高的场景，如网页浏览、邮件、文件传输。
- **UDP：** 适合对实时性要求高的场景，如视频会议、直播、在线游戏、DNS 查询。

## TCP/IP 三次握手的流程是？为什么是三次？

TCP 三次握手的目的不仅是建立连接，更重要的是交换双方的初始序号并确认收发能力正常。

**三次握手流程：**

```
客户端                         服务器
  |  —— SYN, seq=X ————————>     |
  |  <— SYN+ACK, ack=X+1,seq=Y  |
  |  —— ACK, ack=Y+1 ————>      |
  |                              |
  |  ←——— 双方进入 ESTABLISHED — |
```

- 第一次：客户端发 SYN，带上序号 X。
- 第二次：服务器回 SYN+ACK，确认 X 并声明自己的序号 Y。
- 第三次：客户端再发 ACK，确认 Y。

这样双方进入 ESTABLISHED 状态，连接正式建立。

**三次的意义在于：**

1. 用最少的通信次数保证双方都确认了彼此的收发能力。
   第三次握手：客户端回 ACK=Y+1，服务器才确认自己的序号被对方收到，双方序号同步完成。
2. 同时防止过期连接请求坑到服务器。
   有了第三次握手：客户端收到服务器的 ACK，会发现这是个过期请求，立刻回一个 RST，告诉服务器关闭连接，释放资源。

## TCP 四次挥手的过程是怎样的？为什么断开连接要用四次？

### 四次挥手流程

| # | 方向 | 报文 | 含义 | 状态 |
|---|------|------|------|------|
| 1 | 客户端 → 服务器 | `FIN=1, SEQ=U` | 客户端说"我不再发数据了"。 | 客户端 → FIN-WAIT-1 |
| 2 | 服务器 → 客户端 | `ACK=U+1, SEQ=V` | 确认收到了客户端的关闭请求，但自己可能还有数据没发完。 | 服务器 → CLOSE-WAIT，客户端 → FIN-WAIT-2 |
| 3 | 服务器 → 客户端 | `FIN=1, SEQ=W` | 把剩余数据发完后，正式说"我也不发了"。 | 服务器 → LAST-ACK |
| 4 | 客户端 → 服务器 | `ACK=W+1` | 确认服务器的关闭请求。 | 服务器 → CLOSED；客户端 → TIME-WAIT，等待 2MSL 后彻底关闭 |

**为什么断开连接要用四次：**

- TCP 是全双工通信，客户端和服务端停止发送和接受必须分开处理。
- 第 2 次挥手时，服务器收到客户端的 FIN 后必须立即回 ACK，但此时可能还有数据没发完，不能立刻发起断开，否则会导致数据丢失。因此还需要第 3 次，不能合并。
- 客户端最后进入 TIME-WAIT，等待 2MSL，防止最后一个发送的 ACK 丢失，服务器会重发 FIN。保证服务器能正常关闭。是第四次存在的必要性。

---

## TCP 断开连接为什么一般要让客户端去完成？

TCP 规定谁先发 FIN，最终需要进入 TIME-WAIT，等 2MSL 才释放端口。

- 如果让服务器主动断开，它会积累大量 TIME-WAIT，占满端口导致服务瘫痪。
- 客户端的连接数量很少，就算进入 TIME-WAIT 也只占用几十个端口，代价很低。

> 所以断开连接一般由客户端发起，本质是用分散的客户端资源换取服务器的稀缺端口资源。

## 三次握手的过程可以携带数据吗？

三次握手里，前两次不能带数据，第三次可以。

**原因很简单：没确认身份之前不传输数据。**

- 第一次握手的 SYN 如果能带数据，攻击者可以用垃圾数据把服务器内存打爆，所以必须禁止。
- 第二次握手同理，服务器也还在确认阶段。
- 第三次握手时客户端已经确认服务器状态，双方收发能力都验证过了，所以可以顺便带上第一批业务数据，提高效率。

> 如果想在第一次握手带数据，需要用 TCP Fast Open，用之前的 Cookie 来证明身份。

---

## 第四次挥手为什么要等待 2MSL？

有两个原因：**为了可靠关闭连接 + 清理旧包**，保证可靠性和安全性。

**第一：给连接做保险，防止最后一个 ACK 丢失。**

如果 ACK 丢了，服务器会重发 FIN，客户端必须留着补发 ACK，确保服务器能正常关闭。2MSL 刚好覆盖 ACK 和 FIN 在网络中的最大往返时间。

**第二：给网络做清洁，避免旧连接的残留数据包污染新连接。**

等 2MSL 就能保证旧连接的数据包全部自然过期，新连接不会被干扰。

## TCP 是如何保证数据不丢失的

TCP 是个强迫症：每个字节都有逻辑序号、必须被确认、没确认就重传，再配合流量控制、拥塞控制和校验和，最终实现可靠传输。

**TCP 通过多套机制协同保证数据不丢：**

| # | 机制 | 作用 |
|---|------|------|
| 1 | 序列号和 ACK | 确保字节都被确认 |
| 2 | 超时重传和快速重传 | 没确认就补发 |
| 3 | 流量控制 / 拥塞控制 | 保护接收方 / 保护网络 |
| 4 | 校验和 | 保证数据没被传坏 |

> 这些机制叠加在一起，让 TCP 成为可靠传输协议。

## Cookie 和 Session 的作用？

HTTP 是无状态的，浏览器通常用 Cookie + Session 来维持会话：

- **Cookie** 存在客户端，带着 SessionID
- **Session** 存在服务器保存用户状态

> 但在移动端更常用 Token，比如 JWT（json web token），不依赖服务器存储，适合分布式和多端登录，也更方便在 APP 里持久化。
