# Java 面试完整内容

## Java 基础

### C++ 和 Java 的区别

C++ 追求的是性能和对底层的掌控权，Java 主要关注开发更快更安全，并且可以跨平台使用。

从四个核心维度看：内存管理 + 指针 + 跨平台 + 继承

**内存管理：**

- C++ 需要自己申请内存，用完自己释放，防止内存泄漏。
- Java 提供了 GC 机制来处理内存，GC 可能会稍微影响业务，但减少了内存泄漏问题的出现。

**指针：**

- C++ 允许直接用指针操作内存地址，如果使用不当可能直接把整个程序挂掉。
- Java 虽然底层也是地址，但表面做好了安全引用，防止内存使用错误。

**跨平台：**

- C++ 直接把源代码处理成机器语言指令，因此在 Windows 和 Linux 这种跨平台之间，由于机器语言指令不同，无法跨平台使用。
- Java 额外引入了一层字节码，有了字节码就可以跨平台使用。

**类继承：**

- C++ 一个类可以多重继承，允许有多个父类，可能会发生逻辑混乱。
- Java 中规定一个类只能单继承，但额外加了接口可以多实现，作为能力的灵活性扩充。

---

### 抽象类与接口的区别和共同点

- **抽象类：** 可以有成员变量、构造方法、普通方法；适合描述"是（is-a）"关系。
- **接口：** 只能定义方法签名，只能有 abstract 方法，无方法体（Java 8+ 可有 default、static 方法，可以有方法体），适合描述"can-do"能力。

**共同点：** 都不能被实例化。

👉 **速答：** 抽象类是方法复用的模板，接口是定义一种类要实现的规范，作为灵活的能力补充。

---

### 接口中引入了 default 方法和 static 方法，请说明它们的区别和使用场景

**default 方法：**

- 接口中可以有带方法体的实现。
- 实现类可以继承或选择性重写。
- 主要用于接口升级时，避免破坏已有实现类。
- 例子：`List.sort()` 就是通过 default 方法实现的。

**static 方法：**

- 属于接口本身，不属于实现类。
- 不能被继承或重写。
- 必须通过 `接口名.方法名()` 调用。
- 例子：`Comparator.comparing()` 就是接口里的静态方法。

> default 方法是给实现类的可选实现，可以继承和重写；static 方法属于接口本身，不能继承，只能通过接口名调用。default 解决接口升级时的兼容性问题，static 提供工具方法。

**回答示例：**

> 接口里的 default 方法就是给实现类一个"兜底方案"。比如接口升级加了新方法，如果没有 default，所有实现类都得改一遍，太麻烦了；有了 default，老代码不用动也能跑。它相当于接口自己带了个默认实现，类可以直接用，也可以自己改掉。

### 多态怎么理解

多态是面向对象的三大特性之一，同一个方法在不同对象上有不同的表现。

本质上是**编译期按父类型检查方法，运行期按子类型执行方法**，这就是多态。

两种形式：

- **编译时多态（静态多态）：** 通过方法重载实现，编译阶段决定调用哪个方法。
- **运行时多态（动态多态）：** 通过方法重写实现，必须是父类型引用指向子类型对象。编译期按父类型检查方法，运行期按子类型执行方法。编译时看左边（声明类型），运行时看右边（实际对象类型）。

**常见的使用就是：**

```java
List<Integer> list = new ArrayList<>();
```

所谓编译看左边，运行看右边。

**真正体现"不同对象不同表现"，要这样看：**

```java
List<Integer> list1 = new ArrayList<>();
List<Integer> list2 = new LinkedList<>();

list1.add(1);
list2.add(1);
```

- 方法名一样：都是 `add(1)`
- 引用类型一样：都是 `List`
- 实际对象不一样：一个是 `ArrayList`，一个是 `LinkedList`
- 内部表现不一样：
  - `ArrayList` 底层是数组，可能触发扩容、拷贝
  - `LinkedList` 底层是链表，改的是指针

### 有了基本类型为什么要包装类？自动装拆箱是什么？频繁装箱有啥影响？

Java 是一个面向对象的语言，但为了保证性能，提供了 8 个基础数据类型。它们不是对象，没有继承 Object，因此也没有额外方法调用。在一些场景下不可用，因此需要包装类型。

**1. 最常见的集合类泛型**

由于 Java 编译器会执行泛型擦除，编译时泛型会擦除为 Object 类。比如 `List<Integer>`，由于 `int` 根本不是 Object 类，这里如果写 `int` 直接编译报错。

**2. 空值的表达**

基本数据类型是有默认值的，比如 `int` 是 0，就无法在业务中准确判断是空值还是默认值。而 `Integer` 可以通过 `null` 表达空状态，防止表达缺失。

**3. 功能性**

封装之后有更多方法，灵活性更好。比如字符串转数字，就可以用 `Integer.parseInt`。

**自动装拆箱：**

- **装箱：** `int` → `Integer`，编译器会在需要时自动插入 `valueOf()`（装箱）。
- **拆箱：** `Integer` → `int`，`xxxValue()`（拆箱）方法。

**影响：**

- **对象创建开销：** 每次装箱都会创建新的包装类对象（除非命中缓存池，如 `Integer.valueOf()` 对 -128~127 范围的缓存）。频繁创建对象会增加 GC 压力，可能出现卡顿。
- **潜在的空指针问题：** 拆箱时如果包装类对象是 `null`，会抛出 `NullPointerException`。

### Integer 的缓存池机制是如何工作的？范围是多少？能否调整？

**缓存池：**

`Integer.valueOf(int i)` 方法会优先从缓存池中返回对象，而不是每次都新建。

**范围：**

默认缓存范围是 -128 ~ 127。

**实现原理：**

在 `Integer` 类内部有一个静态内部类 `IntegerCache`，在类加载时就会预先创建好这个范围内的 `Integer` 对象，并缓存到数组里。

```java
private static class IntegerCache {
    static final Integer cache[] = new Integer[-(-128) + 127 + 1];
    static {
        for (int i = 0; i < cache.length; i++)
            cache[i] = new Integer(i - 128);
    }
}
```

**能否调整：**

可以通过 JVM 参数 `-XX:AutoBoxCacheMax` 调整上限，但上限只能大于 127，只能大不能小，下限固定为 -128。

### final、static、synchronized

- **final：** 修饰类（不可继承）、方法（不可重写）、变量（不可修改）。
- **static：** 修饰的变量或方法属于类本身，不依赖实例。可以修饰内部类，顶层不行。
- **synchronized：** 保证线程安全，方法或代码块加锁。

`final` 表示一种不可变的状态。`static` 表示变量或方法的归属属于定义的类或接口本身，先于实例对象存在，因此不能用 `this`、`super`。仅 `static` 修饰的内容是可以被修改的。

👉 **速答：** final 限制、static 属于类、synchronized 保证并发安全。

---

### String、StringBuffer、StringBuilder

- **String：** 不可变，线程安全。
- **StringBuffer：** 可变，线程安全（加锁）。底层是一个可变的 `char[]`，带有容量扩展机制。内部方法大多用 `synchronized` 修饰，保证多线程安全。修改时直接在原数组上操作，不会频繁创建新 String 对象。
- **StringBuilder：** 可变，非线程安全，效率高。底层同样是可变的 `char[]`，扩展机制和 `StringBuffer` 一样，就是不加锁，单线程效率高。

👉 **速答：** String 不可变，Buffer 安全，Builder 快。

---

### String 不可变的原因

底层是一个 `final char[]`，不可变。每次修改都会生成新的对象，原数组不变。好处：不可变性保证了线程安全，适合作为常量池、哈希表的 key。

- **安全性：** 避免被篡改（如网络地址、类加载）。
- **性能：** 字符串常量池复用。
- **线程安全：** 不可变对象天然安全。

👉 **速答：** String 不可变是为了安全、性能和线程安全。

### `new String("ABC")` 创建了几个对象？

主要差异需要看字符串常量池是否包含了"ABC"这个对象。

- 如果常量池中已有"ABC"，创建 1 个对象（堆上的 String 对象）。
- 如果常量池中不存在"ABC"，创建 2 个对象（常量池中的 "ABC" + 堆上的 String 对象）。

### equals 与 == 的区别

- **==：** 比较引用地址是否相同。如果是基本数据类型则直接比较数值。
- **equals：** `Object.equals()` 默认也是比较引用地址。但一般会重写用于比较内容是否相同。
- **hashCode：** 对象的哈希值，用于集合（HashMap、HashSet）。

### 如何重写 equals() 方法？为什么还要重写 hashCode()

**为什么重写 equals：**

`Object` 类提供的 `equals` 方法比较的是两个对象之间的引用地址。但实际业务中，我们更关心两个对象的属性值是否相等。因此重写 `equals` 一般发生在自定义一个类对象后，判断该类的实例对象之间是否相等的关系。

**具体实现：**

> 重写 equals() 的过程中要先判断是否同一个对象，再判断是否属于同一个类，最后逐一比较关键字段。其中前两步相当于性能优化，快速帮助比较。

```java
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (o == null || getClass() != o.getClass()) return false;
    Person person = (Person) o;
    return age == person.age &&
           Objects.equals(name, person.name);
}
```

如果属性字段比较多，最终的整体逻辑就会比较重。

因此在 Java 的一些集合类的实现中，比较两个对象时，通常都会先根据传入对象的 `hashCode` 比较。但在 Java 中，**如果两个对象相等，那么它们的 hashCode 就一定相同**。反过来如果两个对象的 hashCode 相同，也可能只是"哈希碰撞"。

所以需要注意：**如果我们重写了 equals 方法，那么就要注意 hashCode 方法，一定要保证能遵守上述规则。**

**重写 hashCode() 方法的核心就是：** 保证和 equals() 一致，同时尽量减少哈希冲突。

📌 **重写原则：**

1. **一致性：** 如果两个对象 equals() 返回 true，它们的 hashCode() 必须相等。
2. **稳定性：** 对象在生命周期中，只要关键字段不变，hashCode() 返回值必须一致。
3. **高效性：** 哈希值分布要尽量均匀，减少冲突。

**常见写法：**

- 推荐用 JDK 提供的工具类：`Objects.hash`，把对象的字段传入即可。

```java
@Override
public int hashCode() {
    return Objects.hash(name, age); // 自动组合字段生成哈希
}
```

👉 **速答：** == 比地址，equals 比内容，hashCode 保证集合查找。重写 equals 后重写 hashCode，为了优化集合类的使用。否则可能集合行为异常。

### 深拷贝与浅拷贝的底层原理是什么？区别在哪里？

- **浅拷贝：** 复制对象引用，指向同一内存。
- **深拷贝：** 复制对象内容，生成独立副本。

👉 **面试速答：** 浅拷贝共享引用，深拷贝独立对象。

**Object 的 clone 在底层是一个 native 原生的方法**，调用的时候就在内存开一块新空间，把老对象的数据每一位复制过去。

这时候，如果复制的是基本数据类型，相当于直接复制了数值；但如果是引用数据类型，相当于复制了一份地址，此时新旧对象共用同一块数据，没有做到拷贝的独立。

**因此要实现深拷贝的方法：**

1. 嵌套式的重写，每个引用字段都克隆一遍，不推荐。
2. 利用序列化和反序列化来拷贝，相当于把对象拆开重新再构建一个。

---

### 序列化和反序列化的作用

序列化是把对象转成字节流，用于持久化、网络传输和深拷贝；反序列化是把字节流还原成对象。

**底层原理：**

1. 序列化是用 `ObjectOutputStream` 把对象写成字节流。
2. 反序列化是用 `ObjectInputStream` 把字节流读回来。
3. 并通过反射重建对象。因此能实现深拷贝。

### Error 与 Exception

- **Error：** 严重错误，属于 JVM 虚拟机层面，程序无法恢复（如 `OutOfMemoryError`、`StackOverflowError`）。
- **Exception：** 可捕获处理的异常。

**受检异常（Checked Exception）：**

- 编译器强制要求处理（try-catch 或 throws）。
- 例如：`IOException`、`SQLException`。
- 特点：程序可预期，必须显式处理。

**非受检异常（Unchecked Exception）：**

- 继承自 `RuntimeException`。
- 编译器不强制处理。
- 例如：`NullPointerException`、`ArrayIndexOutOfBoundsException`。
- 特点：通常由程序逻辑错误引起。

👉 **面试速答：** Error 不可控，Exception 可处理。

---

### Java 中异常分哪两类？有什么区别？

**受检和非受检。**

- **受检异常：** 是由于环境不确定导致的意外，比如文件不存在、网络中断。编译器会审查，并要求提前准备好降级方案（try-catch 捕获）。
- **非受检异常：** 是逻辑不严密导致的低级错误，比如空指针、数组越界，靠代码审查和测试来消灭它。

### 反射机制？它会带来哪些性能开销？如何优化？

- **定义：** 运行时动态获取类信息、调用方法、访问属性。
- **应用场景：** 框架（Spring、Hibernate）、序列化、动态代理。

**获取 Class 对象的三种方式：**

1. **`Class.forName("类全名")`**
   - 传入类的全限定名（包名 + 类名），返回对应的 Class 对象。
   - 常用于框架加载配置里写死的类名，比如 JDBC 驱动：
     ```java
     Class.forName("com.mysql.cj.jdbc.Driver");
     ```
2. **对象.getClass()**
   - 通过已有对象实例获取其运行时的 Class 对象。
     ```java
     String s = "hello";
     Class<?> clazz = s.getClass();
     ```
3. **类名.class**
   - 直接通过类字面量获取 Class 对象。
   - 编译期就能确定，常用于泛型、注解等场景：
     ```java
     Class<String> clazz = String.class;
     ```

**性能瓶颈：**

1. 寻找的过程慢，需要在内存中通过字符串类名查找；JVM 需要进行较多的校验。
2. 反射可以访问和修改类的字段方法，可能造成安全问题。

**优化：**

1. 加缓存，避免每次使用都要搜索。
2. `setAccessible(true)` 可以让 JVM 避开一些权限检查。

👉 **面试速答：** 反射是运行时操作类，常用于框架和动态代理。

### 静态代理与动态代理的区别？JDK 动态代理与 CGLIB 动态代理有何不同

**静态代理：**

- 在编译期就确定代理类，代理类需要手写或提前生成。
- 优点：结构清晰，易于理解。
- 缺点：每个接口都要写一个代理类，代码冗余，扩展性差。

**动态代理：**

- 创建时，通过字节码生成代理对象（JDK / CGLIB 都一样）。
- 运行时，通过反射调用或字节码调用方法。
- 优点：无需为每个接口写代理类，灵活、可复用。
- 缺点：实现复杂，性能略低于静态代理。

👉 **面试速答：**

> 静态代理在编译期就确定代理类，代码冗余；动态代理在运行时生成代理对象，更灵活，常用于 AOP、RPC 等场景。

**JDK 动态代理 vs CGLIB 动态代理的差异：**

- **JDK 代理：** 代理类是生成的，但调用靠反射 → 所以叫反射代理。
- **CGLIB 代理：** 代理类是生成的，而且是自己拼字节码 → 所以叫字节码代理。

### 泛型类型擦除并说说其局限性？为什么不能创建泛型数组？

**泛型在编译期的作用：**

- 编译器会检查类型安全：比如 `List<String>` 只能放 String，放别的会报错。
- 这些信息只存在于编译阶段，保证代码写的时候不出错。

**类型擦除的含义：**

- 到了运行时，JVM 并不保留泛型的具体类型信息。
- 所有泛型参数都会被擦除成它们的**上界**（默认是 Object）。

```java
List<String> list1 = new ArrayList<>();
List<Integer> list2 = new ArrayList<>();
System.out.println(list1.getClass() == list2.getClass()); // true
```

- 运行时它们的类型都是 `ArrayList`，没有区别。

**定义：** 编译时保留泛型，运行时擦除为 Object。

**局限性：**

Java 泛型在编译期有效，运行时类型信息会被擦除成 Object 或上界。这样保证了 Java 版本的向后兼容，但局限是 JVM 运行时不能直接获取泛型类型、反射也拿不到泛型参数。

**不能创建泛型数组的原因：**

类型擦除本身是一个折中的方案。由于 Java 诞生较早，而泛型是后来引进的，因此为了使老版本代码兼容，设定了 JVM 运行时不去关注类型。

而不能创建泛型数组的原因是一个根本的逻辑冲突：数组有运行时类型检查，在运行时需要明确存储类型；而泛型被擦除后运行时类型信息缺失，使用泛型数组可能会导致程序 crash。

👉 **面试速答：** 泛型运行时被擦除，限制了类型操作。

### Lambda 表达式的底层实现原理是什么

核心是**编译器到运行期的能力交接**。

1. 写 `() -> {}` Lambda 表达式。
2. **编译期：** 生成 `invokedynamic` 字节码指令。
3. **运行期：** JVM 执行到这条 `invokedynamic` 指令时，会调用 `LambdaMetafactory` 工厂方法。
4. **工厂**动态生成一个函数式接口的实现类（通常是通过字节码生成）。
5. **最终返回**一个目标方法的引用，完成 Lambda 的运行。

**好处是：**

1. 运行时动态生成对象，不会产生额外的类文件。比每个类单独一个类文件加载开销低。
2. JVM 可以不断优化字节码指令，而我们的代码不用改，就能直接享受优化。

**流程：**

```
1. 写 () -> {} Lambda 表达式
2. 编译期：生成 invokedynamic 字节码指令
3. 运行期：调用 LambdaMetafactory 工厂
4. 工厂动态生成实现类 + 创建实例
5. 绑定方法引用/代码体，完成调用
```

### Java 的注解是什么

类似于**商品的标签和扫码枪**，谁拿扫码枪的很重要！

1. 注解是一种元数据，本质上是一个没有逻辑的标签。
2. 注解的作用主要却决于读取它的对象。
   - 比如编译器用 `@Override` 做语法检查。
   - JUnit 用 `@Test` 做测试识别。
3. **注解本身只是标签，逻辑在读取者。**

## Java 集合

### 谈谈 List、Set、Map 的区别

- **List：** 有序可重复，可以按下标取值。可能会有多个空值存在，所以为防止空指针，有时候需要额外判断。
- **Set：** 无序不可重复，用来去重。允许存在一个 null。
- **Map：** 键值对存储，key 唯一，value 可重复。要看实现，HashMap 可以有一个 null key，Hashtable / ConcurrentHashMap 不行。

👉 **大白话：** List 像队伍，Set 像集合去重，Map 像字典。

---

### 谈谈 ArrayList 和 LinkedList 的区别

- **ArrayList：** 底层数组，查找快，插入删慢。
- **LinkedList：** 底层是一个双向链表，插入删快，查找慢。

👉 **大白话：** ArrayList 查找快，LinkedList 插入删快。

---

### 请说一下 HashMap 与 Hashtable 的区别

- **HashMap：** 非线程安全，效率高，允许 null。
- **Hashtable：** 线程安全（方法加锁），效率低，不允许 null。
  - 效率低：所有方法都加了 `synchronized`，在单线程或低并发场景下性能差。
  - ConcurrentHashMap 在并发场景下更高效。

👉 **大白话：** HashMap 更常用，Hashtable 已过时。

---

### 谈一谈 ArrayList 的扩容机制？

底层就是一个 `Object[] elementData` 数组。

默认构造函数里并不会立刻分配数组，而是在第一次 `add` 时才分配默认容量 10：

```java
private static final int DEFAULT_CAPACITY = 10;
```

当元素数量超过当前数组容量时，就会触发扩容。**默认容量 10，满了就扩容。扩容规则：新容量 = 旧容量 × 约 1.5。**

本质：新建数组，把旧数据拷过去。

👉 **大白话：** ArrayList 扩容就是数组拷贝，按 1.5 倍扩。

**为什么不用按需扩容？倍数扩容不会有内存浪费吗？**

> ArrayList 不采用按需扩容，因为每次扩容都要复制数组，按需扩容会导致整体插入性能退化到 O(n²)。它采用 1.5 倍扩容，是在减少扩容次数（性能）和避免过度浪费内存之间的折中方案。同时 ArrayList 有懒加载机制，不会一开始就占用内存，只有第一次 add 才分配默认容量 10。但是为了尽量避免扩容带来的消耗，如果一开始就能给出 list 的具体容量最好。

---

### CopyOnWriteArrayList 的实现原理及适用场景？为什么它是"弱一致性"的？

**Volatile + ReentrantLock**

**底层结构：** 内部维护一个 volatile 的数组 `Object[] array`。

**读操作（get / 遍历）：**

- 无锁
- 直接读取当前数组引用
- 因为数组是**不可变快照**，所以读永远安全

**写操作（add / remove / set）：**

- 加 `ReentrantLock` 独占锁
- 复制当前数组 → 修改副本 → 新副本替换原数组引用

**弱一致性：**

是指它这种机制只能保证**最终的一致性**，但不能保证实时的读取和修改一致。体现在当使用迭代器进行遍历的时候，迭代器会保存一个读到数组的快照，而不一定是正在被修改的最新的数组。因为写时复制，本身是为了高并发性能的一种妥协方案，不保证强实时一致性。

因此，它这种写时复制的机制，更适合于**读多写少**的场景。因为大量的写操作引发的"加锁 → 复制 → 替换引用"本身对内存就是一种消耗。

---

### HashMap 的底层实现原理？可能出现的问题是什么？

**底层：** 数组 + 链表 / 红黑树（JDK 8 以后引入）。

**插入：** 先算 hashCode 定桶，再用 equals 定 key。

**查找：** 同样先定位桶，再 equals 精确匹配。

**JDK 8 优化：**

- **链表超过阈值转红黑树（JDK 8 引入）：** 当某个桶里的链表长度超过 8 时，会尝试把链表转成红黑树，以提升查找效率。HashMap 在 JDK 8 里，链表长度超过 8 就转红黑树，这个数字是性能权衡的结果：小于 8 用链表更快，超过 8 用树更合适。源码里也写了树节点比链表节点大，所以只有在链表够长时才值得转树。再加上概率分布分析，超过 8 的情况本来就很少，树化只是防止极端碰撞。
- **前提条件：** 整个数组长度要达到 64 以上才会真正转树，否则先扩容数组。由于哈希运算依赖于数组长度，扩容可以让元素分散，降低碰撞概率。
- **退化条件：** 如果树节点数量减少到 6 以下，会退回链表结构。

👉 **大白话：** HashMap 用 hashCode 定桶，equals 定 key。

**扩容时可能发生死循环（JDK 7 修改，JDK 8 已修复）：**

📊 **具体表现：**

当容量不足时，会进行扩容，把旧数组的元素重新分配到新数组。默认 capacity=16，loadFactor=0.75。

- 在**JDK 7 的 HashMap** 中，链表迁移时采用**头插法**。每次迁移一个节点时，都把它插入到新桶链表的**头部**。
- **多线程同时迁移时**，可能导致链表节点指针错乱，形成**环形链表**。后续调用 get() 或迭代时，就会陷入**死循环**，CPU 飙高。

📌 **JDK 8 的改进：**

- JDK 8 改为**尾插法**迁移，避免了环形链表问题。
- 同时引入了**红黑树**结构，优化了高冲突场景下的性能。
- 但注意：HashMap 本身仍然**不是线程安全的**，在并发场景下依然可能出现数据丢失或覆盖，只是不会再出现死循环。

### 请简述 LinkedHashMap 的工作原理和使用方式

在 HashMap 基础上，在每个节点内部就嵌入了双向链表指针，把所有节点串联起来。

```java
static class Entry<K,V> extends HashMap.Node<K,V> {
    Entry<K,V> before, after; // 双向链表指针
    Entry(int hash, K key, V value, Node<K,V> next) {
        super(hash, key, value, next);
    }
}
```

可以看到：

- 它继承了 `HashMap.Node`，所以仍然有 key、value、next（用于哈希桶链表）。
- 额外增加了 `before` 和 `after` 指针，用于维护插入或访问顺序的双向链表。
- 也就是说：每个节点同时存在于两条链上：
  - 一条是 HashMap 的桶链（用于哈希查找）。
  - 一条是 LinkedHashMap 的双向链表（用于顺序维护）。

**额外维护了一条双向链表来记录元素的顺序。**

```
          ┌───────────────────────────────┐
          │       Hash Table (数组)       │
          │  [0] [1] [2] [3] [4] [5] ... │
          └───────────────────────────────┘
                     │     │     │
                     ▼     ▼     ▼
                 ┌────┐ ┌────┐ ┌────┐
                 │Key1│ │Key2│ │Key3│  ← 每个桶指向一个节点
                 └────┘ └────┘ └────┘
                     │     │     │
                     ▼     ▼     ▼
        ┌──────────────────────────────────────────────┐
        │         双向链表（维护插入或访问顺序）        │
        └──────────────────────────────────────────────┘
        Head → [Key1,Value1] ⇄ [Key2,Value2] ⇄ [Key3,Value3] → Tail
```

**常用场景：** 缓存，比如 LRU。

👉 **大白话：** LinkedHashMap 就是 HashMap + 链表，用来保持顺序，做缓存很方便。

---

### 谈谈对 ConcurrentHashMap 的理解

**数组 + 链表 / 红黑树**

**背景：为什么要有它**

- HashMap 在多线程下不安全，Hashtable 虽然安全但效率太低（整张 Map 加锁）。
- 所以出现了 ConcurrentHashMap：既要线程安全，又要高并发性能。

**JDK 7 实现：分段锁（Segment）**

- 整个 Map 被分成若干段（Segment），每段都是一个小 HashMap。
- 每个 Segment 有自己的锁，线程操作不同段时可以并行。
- 优点：锁粒度已经比 Hashtable 小很多。
- 缺点：结构复杂，Segment 数量固定，扩容不灵活。

**JDK 8 实现：CAS + synchronized**

在 JDK 8 的 ConcurrentHashMap 里，插入流程是这样的：

1. **定位桶：** 通过 hashCode 算出数组下标，找到对应的桶（数组的一个槽位）。
2. **CAS 插入头节点：** 如果这个桶是空的（null），就用 CAS（Compare-And-Swap）直接把新节点放到桶的头节点位置。这样不用加锁，效率很高。
3. **冲突时加锁：** 如果桶里已经有节点（链表或红黑树），CAS 就失败了，这时才会用 `synchronized` 锁住这个桶的头节点，保证只有一个线程能安全地修改这个桶里的结构。
4. **后续操作：** 在锁保护下插入新节点，或者在红黑树里做平衡调整。

**性能与安全性：**

- 读操作几乎无锁，写操作锁粒度极细。不会出现死锁，也不会抛出 `ConcurrentModificationException`。
- 适合高并发场景，比如缓存、统计计数、线程安全的共享数据结构。

👉 **大白话：**

> ConcurrentHashMap 是高并发下的安全 Map。JDK 7 用分段锁，每段独立加锁；JDK 8 改成 CAS + synchronized，锁更细，读几乎无锁，效率更高。它就是在多线程里既安全又快的 HashMap。

---

### ConcurrentHashMap 的扩容和 HashMap 相比呢

- **HashMap：** 扩容阈值是 capacity × loadFactor（默认 capacity=16，loadFactor=0.75），超过就扩容一倍数组。
- **ConcurrentHashMap：** 同样默认 capacity=16，loadFactor=0.75，但实现更复杂：
  - 它在多线程环境下，扩容不是单线程完成，而是**多个线程协作**。
  - 每个线程会分配一段桶区间来搬迁数据，避免全表阻塞。
  - 阈值计算和 HashMap 类似，但扩容过程更精细，保证并发安全。

同时，如果正在扩容，会在原数组的桶位置放一个**转发节点**（ForwardingNode），是一个哈希值为负数的节点。此时，如果另一个线程读取到这个节点，会先放弃加入，协助扩容。

**关键区别：**

- **HashMap：** 单线程扩容，期间可能阻塞，读写都受影响。
- **ConcurrentHashMap：** 多线程协作扩容，读操作基本不受影响，写操作只锁桶头。
- **阈值本质：** 两者都是基于 capacity × loadFactor，只是 ConcurrentHashMap 在并发场景下做了更复杂的控制。

---

### 为什么 ConcurrentHashMap 不允许 Key 或 Value 为 null？

核心是为了**消灭二义性**，为了保证结果的唯一性和操作的原子性。

如果允许 null，那么 `map.get(key)` 返回 null 时，就无法区分是**key 不存在**还是**key 存在但 value 为 null**。

ConcurrentHashMap 提供了 `putIfAbsent`、`computeIfAbsent` 等原子操作。如果允许 null，这些方法的判断条件就会失效（因为无法区分"没值"和"值就是 null"）。

---

### ConcurrentHashMap 的 `size()` 方法是如何保证高并发下的统计准确性的？

核心是**用一定的准确性差异，换取高并发下的性能提升**。

具体是维护一个常规数组和临时数组：

- **基础计数（baseCount）：** 默认修改目标，低并发时直接更新。
- **计数数组（CounterCells）：** 高并发下的分散槽位，线程更新失败时退到这里，随机放一下。
- **统计结果：** `size()` 就是 `baseCount + CounterCells` 求和。

**不是实时严格准确的，用近似值换取高并发性能。**

---

### ConcurrentHashMap 能保证完全线程安全吗？

ConcurrentHashMap 的**单个操作是线程安全的**，但**复合操作（如 get + put）不是原子性的**，可能在并发下出问题。

解决方法是用 `putIfAbsent`、`computeIfAbsent` 等原子方法。

---

### ArrayBlockingQueue 与 LinkedBlockingQueue 有何区别？怎么选？

**本质差异来自：数组是固定内存 + 单锁；链表是动态内存 + 双锁。**

- **如果追求高执行效率，且对内存抖动敏感，选 ArrayBlockingQueue。** 高性能是固定容量 + 内存稳定 + 操作开销小（单锁，出入队共用）。
- **如果有高并发的读写需求，能接受一点内存开销，选 LinkedBlockingQueue。** 总的来说 Link 更适合标准的"生产者-消费者"模式。

---

### PriorityQueue 的底层数据结构是什么？它是如何维持堆属性的？

**数据结构：**

本质上是维护了一个动态数组，会扩容，而且扩容逻辑和 ArrayList 几乎一样。

**处理逻辑：**

利用数组的位置维护了一个完全二叉树。对于每个节点，将它下标 i 计算 `2i+1 / 2i+2`，即为它的两个孩子节点。

**维持堆属性：**

默认的 Java 实现是一个**小顶堆**，把最小的元素放在首位。

- **入队：** 首先将新元素放在队尾，然后不断与它的父节点比较大小，直到确定位置。可以理解为入队是一个逐级提拔的过程。
- **出队：** 直接将队首（堆顶元素）出队。然后把队尾的元素填补到队首，再逐渐比较下沉。可以理解为出队是一个"关系户"被逐级开除的过程。

---

### DelayQueue 的实现原理及其在定时任务中的应用

**数据结构与排序逻辑：**

- DelayQueue 本质上是一个**基于 PriorityQueue 的无界阻塞队列**。
- 队列里的元素必须实现 `Delayed` 接口，包含一个过期时间。
- 排序规则：按照过期时间升序排列，最早过期的任务在队首。

**Leader-Follower 模式：**

- **取任务的线程：** 只有一个线程会被 DQ 选为 leader，根据队首元素的过期时间设置定时。
- **其他线程：** 全部进入等待状态（睡眠），避免无意义的轮询。
- **好处：** 降低 CPU 消耗，避免多个线程同时竞争，避免轮询压力。如果用普通队列，需要不断检查是否过期，系统压力大。DelayQueue 通过 Leader-Follower 模式，只在 leader 结束后，接力唤醒下一个。

**应用场景：**

- **订单超时自动取消：** 把订单任务放入 DelayQueue，设置过期时间为"下单时间 + 超时时间"。后台设置一个线程，等着到期后自动触发取消逻辑。
- **定时任务调度：** 比如缓存过期清理、延迟消息队列。

---

### TreeMap 和 TreeSet 的底层原理是怎样的

1. **红黑树的数据结构：染色 + 旋转，保证了左右子树的高度不会差别过大。**

   首先二叉搜索树会有一个问题：如果不对元素加以限制，可能退化成一条链表。比如持续输入 1-2-3-4-5，会退化成一条只有右子树的链表，失去了树的搜索优势。

   ```
         (30,"C")(B)
        /           \
   (20,"A")(R)     (40,"D")(R)
       /                   \
   (10,"X")(B)           (50,"Z")(B)
   ```

2. **TreeSet 本质就是套皮了一层 TreeMap：** 把存放的数据作为 key，定义一个占位对象作为 value。这样利用 key 不能重复、不能为空的机制，就实现了 TreeSet 的功能。

## Java 多线程

### 为什么要使用多线程

1. 线程是 CPU 调度执行的最小单位，管理成本低于进程。
2. 现代系统中业务中存在大量并发，多线程可以提高可用性。

---

### 线程安全怎么理解？有哪些实现方式？

多个线程同时访问某个资源时，程序仍能满足 JMM 的三大语义：**原子性、可见性、有序性**，从而保证最终结果的正确性。

实现方式：

- **原子变量：** 保证对变量的操作是原子的。
- **加锁同步：** 保证资源只能被一个线程访问。
- **不可变对象：** 不允许修改肯定是线程安全的。

---

### Java 的多线程开发中，线程数是不是越多越好？一般怎么设置？

首先从**内存角度**：一个线程大约占 1MB，所以如果大量开可能造成 OOM。

然后从 **CPU 角度**：CPU 需要维护进程的上下文，大量的线程对于 CPU 来说要维护的资源太多，压力也很大。

因此设置线程数需要跟**任务类型**相关：

- **CPU 密集型：** 需要大量计算的情况，一般根据 CPU 核数量 + 1。
- **IO 密集型：** 比如大量读写、网络请求。这种情况一般会有线程在等待任务执行。所以可以多开一些，让 CPU 在其他线程等待 IO 时继续执行别的任务。粗略是 2 倍数量的线程，核心理念是根据等待时间，尽可能把这个等待的空窗期填满。

---

### Java 中什么是线程池？

线程是一套复用和管理线程的机制，可以减少频繁创建销毁线程的开销，有效地复用资源。

**核心矛盾是：** 资源优先，任务是无限的，所以需要线程池作为资源复用和流量缓冲找平衡。

---

### 线程池的核心参数有哪些

- **核心线程 = 正式员工：** 常驻岗位，无论是否有业务，始终在线。
- **阻塞队列 = 排队区：** 核心员工忙碌时，客户进入排队区拿号；根据阻塞队列的类型也会有不一样操作。
- **最大线程 = 临时窗口：** 高峰期开启，缓解排队压力。
- **KeepAlive Time = 下班机制：** 高峰过后，空闲临时窗口必须下班，销毁冗余线程，释放资源。
- **拒绝策略 = 铁面保安：** 当正式员工、临时窗口和排队区都满了，保安决定如何处理：拒绝、丢弃、抛异常或交给调用者。

---

### Java 线程池的拒绝策略有哪些

**奶茶店类比版拒绝策略：**

- **AbortPolicy = 直接拒绝顾客（默认操作！）**
  店里人满了，保安直接说："不接单了！" → 抛异常。

- **DiscardPolicy = 悄悄丢掉订单**
  店员发现太忙，直接把顾客的订单单子撕掉，不通知顾客。

- **DiscardOldestPolicy = 丢掉最老的订单**
  店里太忙，把排队最久还没做的奶茶订单丢掉，腾位置给新顾客。

- **CallerRunsPolicy = 顾客自己做奶茶**
  店员太忙，保安说："你自己去后厨做吧！" → 任务由调用线程自己执行。保证每个都执行。

---

### volatile 关键字的作用是什么

`volatile` 是用来修饰变量的一个关键词，它的作用是：

1. **保证变量的可见性：** 表明这个变量是共享的，每次访问的时候需要到主存读取。
2. **防止指令重排序：** 读取变量的时候会插入特定的内存屏障，保证前后指令的特定顺序。

---

### 如何理解 JMM（Java 内存模型）？

JMM 是一套规范多线程下编程的规则。

1. 抽象出了线程和主内存之间的关系（主内存 - 本地内存）。
2. 规定了 Java 源代码到 CPU 指令需要遵守的编程规范（happens-before）。
3. 定义了原子性、可见性、有序性三大并发语义。

所有并发工具（volatile、synchronized、Lock、原子类）本质上都是在实现 JMM 的语义。

---

### Java 中使用多线程的方式有哪些？/ 启动一个线程的方式有哪些

启动线程的方式主要有四种：

1. **继承 Thread 类：** Thread 类本身就实现了 Runnable。
2. **实现 Runnable 接口。**
3. **实现 Callable 接口配合 FutureTask：** FutureTask 本质上也是一个实现 Runnable 接口的类，用来接收结果。
4. **使用线程池。**

它们本质上最终都是调用 `Thread.start()` 来启动线程，只是任务的封装方式不同。

### 请谈谈 Thread 中 run() 与 start() 的区别？

**run() 方法**最初定义在 Runnable 接口里，表示线程要执行的任务逻辑。Thread 类实现了 Runnable，因此也有 run()。即使不传入接口 Runnable 实现对象也可以运行。

如果你选择继承了 Thread 并重写 run()，那就会执行你写的 run 逻辑；如果是实现 Runnable 接口后传给 Thread，那么调用 `Thread.run()` 会执行实现 Runnable 后重写的 run()。

**Thread.run() 方法：**

```java
@Override
public void run() {
    if (target != null) {
        target.run();   // 如果传入了 Runnable，就执行它的 run()
    }
    // 如果 target == null，就什么都不做
}
```

- **情况 1：** 传入 Runnable → 调用 `target.run()`，执行你实现的任务逻辑。
- **情况 2：** 没传 Runnable → 默认空实现，什么都不做。
- **情况 3：** 继承 Thread 并重写 run() → 覆盖掉这个方法，执行你写的逻辑。

而 **start()** 才是启动新线程，最终由 JVM 调度执行 run()。

### 线程池的状态有哪些

1. **RUNNING → SHUTDOWN：** 调用 `shutdown()`。
2. **RUNNING / SHUTDOWN → STOP：** 调用 `shutdownNow()`。
3. **SHUTDOWN → TIDYING：** 队列任务执行完毕，线程池大小为 0。
4. **STOP → TIDYING：** 所有任务终止，线程池大小为 0。
5. **TIDYING → TERMINATED：** 调用 `terminated()` 方法，线程池彻底结束。

核心线程的停止：SHUTDOWN / STOP 是触发条件，TIDYING 前 `workerCount` 必须变成 0。

### 为什么不推荐使用 `Executors.newFixedThreadPool()`？

**问题核心：**

- `Executors.newFixedThreadPool()` 内部使用**无界队列** `LinkedBlockingQueue`（容量是 `Integer.MAX_VALUE`）。
- 当任务提交速度 > 线程处理速度时，队列会无限堆积，最终可能导致**内存溢出（OOM）**。
- 线程池参数被隐藏，开发者无法灵活配置核心线程数、最大线程数、队列大小和拒绝策略。

**具体风险：**

1. **内存风险：** 无界队列无限增长，可能撑爆内存。
2. **扩展性差：** 线程数固定，无法根据负载动态扩容。
3. **拒绝策略不可控：** 默认 `AbortPolicy`，直接抛异常，不利于业务容错。
4. **参数不可见：** 工厂方法封装了 `ThreadPoolExecutor`，开发者不了解底层配置，难以调优。

**推荐做法：**

直接使用 `ThreadPoolExecutor`，手动配置参数：

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    2,                      // 核心线程数
    4,                      // 最大线程数
    60, TimeUnit.SECONDS,   // 空闲线程存活时间
    new ArrayBlockingQueue<>(100), // 有界队列，避免无限堆积
    Executors.defaultThreadFactory(),
    new ThreadPoolExecutor.AbortPolicy() // 拒绝策略，可根据业务调整
);
```

**一句话：** 不推荐使用 `Executors.newFixedThreadPool()`，因为它内部使用无界队列，任务过多时可能导致 OOM。另外线程池的核心参数不可控，不利于手动定制，比如拒绝策略。另外业务中为了追踪一般会给线程起标识名，内置线程池实现不了。推荐直接用 `ThreadPoolExecutor`，手动配置线程池参数。

### 说一下线程的几种状态？

**速答版：**

> Java 线程有六种状态：NEW、RUNNABLE、BLOCKED、WAITING、TIMED_WAITING、TERMINATED。RUNNABLE 包含就绪和运行，BLOCKED 是等锁，WAITING / TIMED_WAITING 是等待，最后 TERMINATED 表示结束。

- **NEW：** 新建状态，线程对象已创建但还没调用 `start()`。
- **RUNNABLE：** 就绪 / 运行状态，线程已调用 `start()`，等待 CPU 调度或正在运行。注意：Java 把"就绪"和"运行"统一叫 RUNNABLE。
- **BLOCKED：** 阻塞状态，线程在等待获取锁（synchronized）。
- **WAITING：** 无限期等待状态，调用 `Object.wait()` 或 `join()` 等方法，直到被唤醒。`wait()` 会让线程进入 WAITING，需要其他线程 `notify()` 才能唤醒。
- **TIMED_WAITING：** 有时限的等待状态，比如 `sleep(ms)`、`wait(ms)`、`join(ms)`。`sleep()` 会让线程进入 TIMED_WAITING，时间到自动恢复。
- **TERMINATED：** 终止状态，线程执行完毕或异常退出。

```
NEW --> RUNNABLE --> TERMINATED
          |
          |---> BLOCKED (等待锁)
          |---> WAITING (无限等待)
          |---> TIMED_WAITING (限时等待)
```

---

### 线程 BLOCKED 和 WAITING / TIMED_WAITING 有什么区别？

- **BLOCKED** 是锁竞争失败，被动阻塞。
- **WAITING** 是无限期等待别人唤醒，主动调用 wait 进入。
- **TIMED_WAITING** 是有时限的等待，sleep 或者 wait 一段时间。

**BLOCKED 不释放锁，WAITING / TIMED_WAITING 如果是 wait 会释放锁。**

---

### 谈一谈线程 sleep() 和 wait() 的相同点和区别？

**相同点：**

- 都会让当前线程进入等待状态。
- 都可能导致线程暂时不执行。

为什么用"可能"？因为：

- sleep / wait 一定会改变线程状态。
- sleep / wait 不保证立即停止执行。
- sleep / wait 不保证暂停期间不会被调度器短暂调度。

因此：它们会让线程进入等待状态，并且通常会暂停执行，但暂停行为不是同步保证，所以用"可能"更严谨。

**区别点：**

- `sleep()` 是 Thread 的静态方法，让线程休眠一段时间，需要传入一个时间。**不释放锁**，时间到自动恢复。
- `wait()` 是 Object 的方法，必须在同步块里调用，**会释放锁**，进入等待队列，需要其他线程 `notify()` 才能唤醒。

**sleep 用于"暂停执行"，不释放锁；wait 用于"线程协作"，会释放锁并等待 notify 唤醒。**

- sleep 适合控制节奏，wait 适合等待条件。

就能把状态题和方法区别题串起来：

- 状态题：线程可能进入 TIMED_WAITING 或 WAITING。
- 区别题：sleep 不释放锁，wait 释放锁并依赖 notify。

---

### Java 线程中 notify() 和 notifyAll() 有什么区别？

- `notify()` 唤醒一个等待线程进入 RUNNABLE，具体哪个由 JVM 决定。
- `notifyAll()` 唤醒所有等待线程，让它们重新竞争锁。

前者效率高但不公平，后者公平但可能带来性能开销。

---

### 如何实现多线程中的同步？

- **最基本的是 JVM 关键字 synchronized。**
- **更灵活的是 Lock。**
- **轻量操作可以用基于 CAS 的原子类：** 无锁实现性能高。
- **线程协作可以用并发工具类：** CountDownLatch、CyclicBarrier、Semaphore。

**并发工具类的本质都是基于 AQS，而 AQS 的核心就是 CAS + 队列。** CountDownLatch 用 CAS 递减计数，Semaphore 用 CAS 更新 permit，CyclicBarrier 用锁和条件变量配合 CAS。可以说它们的底层都是靠 CAS 保证原子性。

变量可见性用 volatile。

---

### 对 synchronized 怎么理解？

**作用：** Java 提供的同步机制，保证同一时刻只有一个线程能访问被锁的资源。可以用来修饰方法、代码块。

📌 **使用方式：**

1. **修饰方法** → 实例方法锁当前对象实例。静态方法锁当前类的 Class 对象，所有实例共享。
2. **修饰代码块** → 括号内锁指定对象或类 Class 对象。

👉 **总结：** synchronized 加在实例方法上锁对象实例，加在静态方法或 `synchronized(Class)` 上锁类对象。

**本质上就是基于对象头（Object Header）里的锁标记来实现的。**

具体机制：

- **对象头结构：** 每个 Java 对象在内存里都有一个对象头（Object Header），其中包含 Mark Word，里面存储了标识对象的哈希码、GC 信息、以及锁状态。

📌 **特殊点：** 构造方法不能用 synchronized 修饰，但可以在内部写同步代码块。

---

### synchronized 的原理是什么

**底层原理：**

都是在字节码层面：

- **同步代码块：** 编译成 `monitorenter` / `monitorexit` 指令。进入时加锁，执行后释放。
- **同步方法：** 方法表里有 `ACC_SYNCHRONIZED` 标识。调用时自动加锁。
- **最终实现：** 都依赖对象头 Mark Word 里的锁标记来记录当前锁的不同状态。

对象头里的 Mark Word 只是用来存储锁状态的信息。

👉 **速答版：**

> synchronized 依赖 JVM 字节码。代码块编译成 monitorenter/monitorexit，方法有 ACC_SYNCHRONIZED 标志。实现调用时的加锁。

**Synchronized 通过对象头里的 Mark Word 记录锁状态：**

- 无锁时存哈希码。
- 偏向锁存线程 ID。
- 轻量级锁存 Lock Record 指针。
- 只有在重量级锁时才会指向 Monitor。

---

### 什么是 synchronized 的锁升级

synchronized 的锁会根据竞争情况逐步升级，目的是在保证线程安全的同时尽量提高性能。

**偏向锁（Biased Lock）：**

- 特点：Mark Word 记录第一个获得锁的线程 ID。
- 触发条件：无竞争，单线程反复进入。
- 优势：同一线程反复进入同步块时，无需真正加锁，性能最好。

**轻量级锁（Lightweight Lock）：**

- 特点：Mark Word 指向线程栈中的 Lock Record。
- 触发条件：有多线程竞争，但冲突不严重。
- 机制：线程通过 CAS 尝试获取锁，如果失败会自旋等待。

**重量级锁（Heavyweight Lock）：**

- 特点：Mark Word 存放一个指针，指向 JVM 内部的 Monitor 对象。
- 触发条件：竞争激烈，自旋失败。
- 机制：线程挂起，进入 Monitor 的等待队列，由操作系统调度。

---

### 线程死锁，如何有效地避免线程死锁？

**线程死锁**是指多个线程同时被阻塞了，并且它们一个或多个都在等待某个资源的释放，形成了循环等待，导致无限期阻塞。

**产生死锁的四个必要条件：**

1. 资源互斥
2. 线程已经持有资源不释放，同时又在请求其他资源
3. 其他线程正在使用的资源不可被剥夺
4. 多个线程形成了循环等待

**只要破坏一个条件即可避免。**

---

### 谈谈线程阻塞的原因？

线程阻塞的原因主要有：

- 等待锁
- 等待其他线程
- I/O 阻塞
- 死锁

比如 sleep 不释放锁，wait 会释放锁。join 等待其他线程结束，yield 只是让出 CPU。

---

### synchronized 和 volatile 关键字的区别？

**首先，作用的对象来说：**

- volatile 只能修饰变量。
- synchronized 可以修饰方法、代码块。

**另一方面：**

JMM 描述在并发编程时对共享变量的要求要保证：原子性、有序性、一致性。

**volatile 关键字：**

- 首先可以保证**有序性**，修饰的变量在读写时会在字节码层面加入内存屏障，防止了重排序。
- 其次可以保证**可见性**，JVM 每次读取变量都需要到主存中读取。
- **但是，不能保证原子性！** 比如对一个变量的修改，需要通过"读取 - 操作 - 写回"这三个步骤，volatile 无法保证这个流程的原子性。

**synchronized 都可以保证：** 通过加锁保证了方法块内代码的原子性。并且在进入 / 退出修饰的方法或者代码块的时候，会把本地的缓存数据和主内存中刷新，保证了一致性和可见性。

---

### AQS 速答版（功能补充）

- **AQS：** Java 并发包里的抽象类，用来构建各种同步器（如 ReentrantLock、Semaphore、CountDownLatch、CyclicBarrier）。
- **核心机制：** 基于 CLH 队列 + CAS，线程获取锁失败时进入队列排队，前驱释放锁后唤醒后继。

**应用场景与功能：**

- **ReentrantLock → 独占锁：** 提供比 synchronized 更灵活的锁机制，可重入、可中断、支持公平锁。
- **Semaphore → 共享锁：** 控制并发访问的线程数量，常用于限流或资源池。
- **CountDownLatch → 倒计时器：** 等待一组任务完成后再继续执行，常用于主线程等待子任务。
- **CyclicBarrier → 循环屏障：** 一组线程到齐后一起继续执行，可重复使用，常用于分阶段计算。

### 谈谈 ThreadLocal 用法和原理？

**ThreadLocal 主要是用来实现线程间的数据隔离**，本质上是用私有的副本换取无锁的并发。

每个线程内部维护了一个 `ThreadLocalMap` 用来存储线程的局部变量，作为独立的数据副本，避免共享冲突。key 是 ThreadLocal，value 是副本。这样一来同一个线程就可以创建多个 ThreadLocal 变量，但每个线程访问的都是自己的数据副本，避免共享冲突。

可以用来存储线程全局上下文的一些参数，如用户信息、traceId、事务对象，作为一个公共入口。

**ThreadLocal 为什么会造成内存泄漏？**

主要是由 ThreadLocal 内部维护的 map 的数据结构和 Java 的 GC 机制导致。

Map 的 key 是 ThreadLocal 的**弱引用**，而 value 是**强引用**。在 Java 的 GC 逻辑中，弱引用在没有被外部引用的情况下，GC 中会被认为可以释放。因此会导致 map 中 key 被释放为 null，而 value 仍存在的问题，就可能造成内存泄漏。

**因此使用完 ThreadLocal 后最好手动调用 remove 方法，释放对应的 value 空间。**

```java
// 定义一个线程局部变量
private static ThreadLocal<Integer> local = ThreadLocal.withInitial(() -> 0);

public void doSomething() {
    try {
        local.set(10);   // 给当前线程设置副本
        // 使用逻辑
        int value = local.get(); // 获取当前线程的副本
    } finally {
        local.remove();  // 使用完后清理，避免内存泄漏
    }
}
```

### 什么是 happens-before

**底层逻辑：** 计算机底层非常"鸡贼"，只要不影响结果，顺序随时可能调换！因此不是跟时间顺序相关。

Happens-before 是一套原则，关注的是并发下的**可见性**。A 的结果，B 可见，就可以说 A happens before B。

---

### 什么是 CAS？什么是 ABA 问题？如何解决？

**核心操作：** Compare and Swap，把对比 + 交换打包成一个 CPU 指令级别的原子操作。跳过了复杂的锁申请过程，无锁，直接通过硬件（CPU 指令）的原子性来保证安全。CAS 这种机制本质上就是用极小的检查成本，换取了极大的执行效率。

**ABA 问题**则是 CAS 的典型陷阱：值从 A 变成 B 再回到 A，CAS 误以为没有变化，从而导致逻辑错误。

**危害：** 在指针或内存复用场景（如无锁栈 / 队列）可能导致数据结构损坏或逻辑错误。

**解决方法：** 引入版本号（如 Java 的 `AtomicStampedReference`）。

---

### AtomicInteger 适合高并发，但在竞争极其激烈时性能会下降，为什么？有什么优化方案？

**原因：** 高并发下所有线程都在关注同一个变量，这种单点的竞争导致了严重的内耗。在竞争激烈时，大量线程 CAS 失败 → 不断重试 → CPU 内耗。

**优化方案：**

把 AtomicInteger 换成 **LongAdder**，用一定空间换时间。

- **AtomicInteger：** 所有线程都在 CAS 同一个 value 字段，竞争非常激烈。
- **LongAdder：** 把一个计数拆分成多个 Cell（槽位），每个线程随机选择一个 Cell 来做 CAS 更新。
  - 这样不同线程更新不同 Cell，减少了热点冲突。
  - 最终统计时，把所有 Cell 的值加总起来就是总计数。

**适用场景：** 写多读少，高并发统计，而且并发量极高的计数统计。

## JVM

### Java 是编译型还是解释型？

Java 既不是纯编译型也不是纯解释型，它先编译成字节码，再由 JVM 解释执行，并结合 JIT 编译优化性能，所以是一门**混合型语言**。这也是 Android 虚拟机（Dalvik / ART）设计的核心思想：解释器和编译器配合，兼顾跨平台和性能。

- **表明：** Java 是一门半编译、半解释的混合型语言。
- **实际上是：** 解释器和编译器在一起打配合，实现性能与灵活性的完美平衡。

---

### 什么是 JIT？

**JIT（Just-In-Time Compiler，即时编译器）**是 JVM 在运行时的一个优化机制：它会把字节码中"热点代码"直接编译成本地机器码，从而避免解释执行的开销，大幅提升性能。

**作用：**

- **解释器的缺点：** 逐条解释字节码，灵活但性能低。
- **JIT 的优势：** 发现某段代码被频繁调用，就把它编译成本地机器码，后续直接运行机器码，速度接近 C/C++。

**优化策略：**

- **热点探测（HotSpot VM 会统计方法调用次数）。**
- **内联优化（把小方法直接展开）。**
- **逃逸分析（优化对象分配，减少堆内存压力）。**

---

### 如何理解 JVM 的内存逃逸（Escape Analysis）？

**逃逸分析（Escape Analysis）：** JVM 在 JIT 编译时会分析对象的作用范围，判断对象是否会"逃逸"出当前方法或线程。

**核心是通过 JVM 的检测，减少不必要的性能开销。**

判断对象是否逃逸出方法或线程。如果对象只在方法内部使用，不会被外部引用，就没必要分配到堆上，可以在栈上分配或进行**标量替换**，从而减少堆内存分配和 GC 开销。

---

### JVM 中堆和栈有什么区别？

**核心逻辑：** 执行与存储、私有与共享。只要抓住了执行与存储、私有与共享这两对核心矛盾，就能清晰区分。

### Java 对象的内存布局是怎样的？

**一句话开场：**

在 JVM 中，一个对象的内存布局主要分为三部分：**对象头、实例数据、对齐填充。**

**1. 对象头（快递单）：**

- **标记字（Mark Word）：** 存储对象的哈希码、GC 分代年龄、锁状态。面试常考：锁的状态记录（偏向锁、轻量级锁、重量级锁）。
- **类指针（Class Pointer）：** 指向对象所属的类元数据，告诉 JVM 这个对象是什么类型。
- **数组长度（仅数组对象有）：** 记录数组的长度信息。

**2. 实例数据（盒子里的物品）：**

- 存放代码里定义的字段（如姓名、年龄、订单号）。
- JVM 会进行**字段重排序**：大字段（long、double）放在一起，小字段（byte、boolean）放在一起。目的：节省空间，提高访问效率。

**3. 对齐填充（泡沫填充物）：**

- HotSpot 要求对象起始地址必须是 8 字节的倍数。
- 如果实例数据不足，会填充一些无意义的字节。
- 目的：让 CPU 读取更高效，因为 CPU 习惯按块读取。

📊 **总结逻辑：**

- 对象头 = 快递单（标记字 + 类指针 + 数组长度）
- 实例数据 = 商品本身（字段，经过重排序）
- 对齐填充 = 泡沫填充物（保证 8 字节对齐，提高 CPU 效率）

✅ **面试速答公式：**

> Java 对象在内存里分为三部分：对象头（记录哈希码、锁状态、类信息）、实例数据（字段，经过重排序）、对齐填充（保证 8 字节对齐）。对象头是重点，尤其是 Mark Word 的锁状态。

其中，Mark Word 的 64 位是动态复用的：

- 无锁时存哈希码和 GC 年龄。
- 偏向锁时存线程 ID。
- 轻量级锁时存 Lock Record 指针。
- 重量级锁时存 Monitor 指针。
- 哈希码如果和锁信息冲突，就搬到 Monitor 或 Lock Record 保存。

### JVM 的运行内存区域有哪些？

**核心理解：共享 2 + 私有 3**

共享为程需要共用的场景、存放全局性数据（对象、类信息）；私有为线程独享、一般是运行时操作相关。

- **堆（Heap）：** 最大的仓库，存放对象实例，垃圾回收器的主要战场。
- **方法区（Method Area）：** 存放类信息、常量、静态变量。Java 8 后改为元空间（Metaspace），搬到本地内存。
- **Java 栈（Java Stack）：** 每个线程独享，方法调用时生成栈帧，存局部变量、操作数栈、返回地址。递归过深会导致 `StackOverflowError`。
- **本地方法栈（Native Method Stack）：** 类似 Java 栈，但用于执行 C/C++ 写的 native 方法。
- **程序计数器（PC Register）：** 线程私有，保存当前执行的字节码指令地址，保证线程切换后能恢复执行。**唯一不会发生内存溢出的区域。**

🎯 **速答：**

> JVM 内存区域分为共享的堆和方法区，以及线程私有的 Java 栈、本地方法栈和程序计数器。堆存对象，方法区存类信息；栈存方法调用信息；程序计数器记录当前执行位置。Java 8 后方法区改为元空间，搬到本地内存以避免溢出。

---

### 永久代为什么要改成元空间？

**永久代（PermGen）：** 在 JDK 7 及之前，类信息存放在堆空间里人为划出的一个固定小房间。

**缺点：**

- 大小固定，无法扩展；框架动态生成类时容易爆仓 → `OutOfMemoryError: PermGen space`。
- 调优困难：设大浪费，设小容易溢出。

**元空间（Metaspace）：** JDK 8 开始，把类信息搬到操作系统的**本地内存**。

**优点：**

- 不再受限于 JVM 堆空间，可以根据物理内存自动扩展。
- 依然可以设置上限，但默认更灵活、更健壮。

📊 **逻辑：**

> 永久代的问题在于容量固定，无法适应现代框架动态生成大量类的需求。JDK 8 改为元空间，把类信息存到本地内存，本质上解决动态增长和固定容量之间的矛盾。目的就是打破空间的枷锁，让系统更灵活、更稳定。

---

### 字符串常量池为什么搬到堆内存？

在 Java 6 及之前，它在方法区的**永久代**里，问题在于：永久代的空间是固定的，无法扩展，容易 OOM。

从 **Java 7 开始**，字符串常量池被移到堆里，这样 GC 可以统一管理，空间也更灵活。

到了 **Java 8**，PermGen 被彻底取消，用 Metaspace 替代，但字符串常量池依然留在堆中。

### JVM 类加载的过程是怎样的？

**1. 加载（Loading）**

- JVM 根据类的全限定名去磁盘找 .class 文件。
- 把字节码读入内存，生成一个代表该类的 Class 对象，作为访问入口。
- 类比：把纸质图纸扫描成电子档，录入系统。

**2. 连接（Linking）**

- 分为三步：
  - **验证（Verification）：** 检查字节码格式、逻辑合法性，防止非法代码。
  - **准备（Preparation）：** 为类的静态变量分配内存并赋默认值。
  - **解析（Resolution）：** 把常量池里的符号引用替换为直接的地址引用。
- 类比：录入系统后，先查毒、再分配资源、最后把图纸里的符号都指向真实对象。

**3. 初始化（Initialization）**

- 执行类构造器 `<clinit>` 方法，给静态变量赋初始值。
- 类比：施工图纸正式生效，开始投入使用。

📊 **逻辑：**

> 类加载就是 JVM 把磁盘上的字节码文件变成内存里的 Class 对象。它分为三个阶段：加载（读入字节码生成 Class 对象）、连接（验证、准备、解析）、初始化（执行 `<clinit>` 方法）。本质上就是从冷冰冰的字节码到活生生的类对象的过程。

### 双亲委派的过程是怎样的？目的是什么？

**8 个字总结：向上委托，向下加载。**

**流程：**

1. 当类加载器接到加载请求时，不是自己直接干，而是先交给上层加载器。
2. 一层层向上委托，直到最顶层的**启动类加载器**。
3. 如果顶层能加载，就直接返回；如果不能，再逐层往下传，最终由底层加载器完成。

**本质目的：**

- **防止核心类被篡改：** 比如有人写了一个假的 `java.lang.String`，双亲委派保证最终还是由启动类加载器加载正版。
- **保证系统安全与一致性：** 所有类加载请求先上报，只有顶层说"不归我管"，才会逐层下放。

### 什么情况下需要打破双亲委派？

双亲委派的核心是"向上委托，向下加载"，保证安全和秩序。但它的缺点是**可见性单向**：上层看不到下层的类。于是，在某些场景下必须打破规矩。

**两个经典场景：**

**1. JDBC 驱动加载（JDK 和 Tomcat 自己选择的解决方案）**

- 核心接口定义在 JDK 的核心包 rt.jar，由启动类加载器负责。
- 具体驱动（MySQL、Oracle）在用户项目路径，由应用类加载器负责。
- 启动类加载器无法直接访问应用类加载器的类。
- **解决方案：** 设置线程上下文类加载器（TCCL），让顶层"借用"底层的能力。`Thread.currentThread().setContextClassLoader`
- **本质：** 这是 SPI 服务发现机制的调用。

**2. Tomcat 等 Web 容器**

- 多项目共存，可能使用不同版本的框架。
- 如果死守双亲委派，新项目 B 会被迫使用老项目 A 已加载的旧版本类。
- **解决方案：** Tomcat 自定义类加载器，方法不去调用父类。先在自己的空间里找，再决定是否委托。
- **本质：** 这是版本隔离与热部署的需求。

**方法实现上：** 自定义一个 ClassLoader，重写 `loadClass()` 方法。让他不要再调用父类。

**逻辑：**

> 双亲委派模型保证了核心类库的安全，但它的可见性是单向的，上层看不到下层。在 JDBC 驱动加载时，核心库需要反向调用用户代码，就必须通过线程上下文类加载器打破规矩；在 Tomcat 容器里，为了实现多版本隔离和热部署，也必须自定义类加载器。这说明双亲委派不是雷打不动的，而是需要安全与灵活之间的权衡。

### JVM 是如何判断一个对象是可回收的？

**1. 早期方法：引用计数法**

- **原理：** 每个对象维护一个引用计数器，有引用时 +1，引用断开时 -1；计数为 0 就认为对象可回收。
- **问题：** 无法解决循环引用。两个对象互相引用，即使外部都不再使用，它们的计数器也不会归零，导致内存泄漏。
- **结论：** 简单但不可靠，现代 JVM 已经淘汰。

**2. 现代方法：可达性分析（Reachability Analysis）**

- **原理：** 从一组特殊的对象集合 **GC Roots** 出发，沿着引用链遍历。
  - 能被 GC Roots 直接或间接关联的对象 → 存活。
  - 完全断开联系的对象 → 可回收。
- **优势：** 避免了循环引用问题，本质是"找关系"。

**3. GC Roots 的典型分类：**

- 局部变量表：方法栈帧里的引用。
- 静态变量：方法区里的类静态成员。
- 常量：方法区里的常量引用。
- 本地方法引用：JNI 调用的对象。

---

### 如何判断一个类可回收？常量是废弃常量？

一个类的对象是否可回收，取决于"对象是否可达"；一个变量是否是废弃变量，取决于"是否仍然存在有效引用链"。

**类的元数据（Class 对象）是否可卸载（Class Unloading）？**

JVM 只有在以下条件**同时满足**时才会卸载类：

1. 类的 ClassLoader 不再被引用
2. 该类的 Class 对象不可达
3. 该类没有实例对象存活

也就是说：**类的卸载是以"类加载器是否可回收"为前提的。** 这就是为什么**系统类加载器加载的类永远不会被卸载。**

**常量是否是"废弃常量"？**

常量池中的常量如果没有任何引用指向它 → 废弃常量 → 可回收。

### Java 中的强引用、软引用、弱引用、虚引用有什么区别？

四种引用的核心区别不是语法，而是 GC 在扫描可达性图时，对它们的"态度强弱"。

**1. 强引用：生死之交**

- 代码里最普通的引用：`Object o = new Object();`
- GC 的态度：绝不回收
- 即使内存要 OOM，也不会动它
- 这是 JVM 默认的引用类型

**一句话：你在，我就在。**

**2. 软引用：好朋友**

- 用 `SoftReference` 包装
- GC 的态度：内存紧张时才回收
- 典型用途：缓存（图片缓存、对象缓存）

**一句话：家里宽裕你住着，家里紧张你先走。**

**3. 弱引用：点头之交**

- 用 `WeakReference` 包装
- GC 的态度：只要 GC 开始工作，就会回收
- 不看内存是否紧张
- 典型用途：WeakHashMap、ThreadLocal Map 的 key

**一句话：只要你来收，我就走。**

**4. 虚引用：灵魂出窍的影子**

- 用 `PhantomReference` + `ReferenceQueue`
- 不能通过它访问对象
- 唯一作用：对象被回收时收到通知，用于管理堆外内存、资源释放

**一句话：我走了会给你留个信。**

**为什么这四种引用存在？**（面试官最爱问）

因为 JVM 的 GC 是基于可达性分析的：不同引用类型，就是不同强度的"可达性"。GC 扫描对象图时，会根据引用类型决定是否把对象加入"可达集合"。

- 强引用：必须加入
- 软引用：内存紧张时不加入
- 弱引用：永远不加入
- 虚引用：永远不加入，但会发通知

### 常见的垃圾收集算法有哪些？

GC 的本质就是在"节省空间"与"提高效率"之间做权衡。

1. **标记-清除：** 省空间但会碎片；适用场景：老年代（对象大、活得久）。
2. **标记-复制：** 快但浪费空间；无碎片，但需要两块等大内存（浪费 50%）；适用场景：新生代（对象朝生夕死）。
3. **标记-整理：** 兼顾两者但最慢。所有活着的对象往一边推，慢但整齐；适用场景：老年代（对象多、不能频繁复制）。

JVM 采用**分代收集**，就是把不同生命周期的对象放在不同区域，用最适合它们的算法处理，从而在时间和空间之间取得最优平衡。

---

### 为什么新生代用复制算法？老年代不用？

因为：

- **新生代对象死亡率极高（80%~98%），** 复制成本极低。
- 所以 JVM 采用 **8:1:1 的 Eden + Survivor 结构：**
  - 80% 放新对象
  - 10% Survivor From
  - 10% Survivor To

只浪费 10% 空间，就换来极高效率。可以说：**复制算法在新生代几乎是"白捡便宜"。**

---

### 什么是 STW（Stop-The-World）？

**Stop-The-World** 是指在垃圾回收过程中，JVM 会暂停所有用户线程，只让 GC 线程运行。

**本质原因是** 一致性：垃圾回收需要在一个静态快照上做可达性分析，才能准确区分哪些对象是垃圾，哪些仍在使用。如果用户线程还在修改引用关系，GC 就可能误判，把正在使用的对象回收掉 → 程序直接崩溃。所以 STW 的本质是：**冻结现场，保证内存快照的一致性。**

**现象：** 程序响应突然中断，用户可能感觉到卡顿或延迟。

**现代 GC 收集器无法彻底消灭 STW，但通过各种手段降低它的影响：**

- **CMS：** 并发标记，减少停顿时间，但仍有短暂 STW。
- **G1：** 把一次长停顿拆成多个短停顿，用户几乎感知不到。
- **ZGC / Shenandoah：** 通过并发和染色指针，把停顿压缩到毫秒级。

**一句话总结：** 现代收集器的目标不是消灭停顿，而是把停顿切碎、缩短，让用户几乎感觉不到。

**这样说：**

> Stop-The-World 是 JVM 在垃圾回收时暂停所有用户线程的现象，本质是为了保证内存快照的一致性。它可能导致程序卡顿、吞吐量下降甚至延迟不可控。现代收集器如 CMS、G1、ZGC 并不能彻底消灭停顿，但通过并发和拆分，把长停顿变成很多极短的停顿，从而在吞吐量和响应时间之间取得平衡。

---

### 常见的垃圾收集器有哪些？

垃圾收集器的演化本质上是围绕**吞吐量**和**延迟**的权衡。

- 最早的**串行收集器**停顿感强，但简单高效。
- **并行收集器**利用多核提升吞吐量。
- **CMS** 通过并发降低停顿，但有碎片问题。
- **G1** 则通过区域化回收和可预测停顿实现平衡。
- **现代收集器如 ZGC** 进一步把停顿压缩到毫秒级。

**调优的核心目标：** 不是消灭停顿，而是让停顿短到用户几乎察觉不到。

---

### G1 对比 CMS 优势在哪？

**CMS 的问题：**

- **算法：** CMS 用的是标记-清除。
- **现象：** 清理后会留下很多零散的小空隙，就像书架上随便塞书，时间久了到处都是碎格子。
- **本质问题：** 虽然总空间够，但没有连续大块，导致大对象放不下。只能触发一次"全场停顿"的整理（Full GC），卡顿非常明显。

CMS 用标记-清除，容易产生碎片，因此常常需要一次长时间的 Full GC。所以有 CMS 的"碎片化 + 停顿不可控"问题。

**G1 的革命在于：** 把堆切成很多小格子，用复制算法避免碎片，并通过收益模型和可预测停顿来控制卡顿时间。它不再死板地分新生代和老年代，而是化整为零，智能规划，让停顿更可控。这就是从 CMS 到 G1 的质变。

---

### GC 频繁有可能是什么原因造成的？

**本质上是内存压力过大：**

- 要么是对象创建太快，年轻代撑不住。
- 要么是老年代被塞满，空间不足。
- 要么是虚拟机参数设置不合理，扩容前拼命清理。

**首先总体上考虑：** 是不是堆内存参数设置不合理，堆内存不足的时候，申请扩容前频繁触发 GC。

**然后从代码逻辑上考虑：** GC 频繁一般是因为对象创建过快或者老年代空间被占满。常见原因有**大对象直接进入老年代**（比如一次性加载百万行数据或超大图片）；还有**内存泄漏**（比如静态集合持有引用，导致对象无法释放）。

**解决思路就是两抓：**

- 一抓大流量，避免不合理的大对象。
- 二抓顽固引用，避免内存泄漏。

---

### JVM 和 Android ART 在 GC 方面的差异有哪些？（有差异的重点）

**垃圾回收器：**

- **JVM：** 考 CMS、G1、ZGC，强调吞吐量和延迟。
- **Android ART：** 考并发复制 GC、增量 GC，强调低停顿，避免 UI 卡顿。

**GC 频繁原因：**

- **JVM：** 大对象进入老年代、内存泄漏、堆参数不合理。
- **Android：** 除了这些，还要强调 Bitmap 占用过大、Context 泄漏、集合类误用 → 直接导致掉帧和 ANR。

**JMM 内存模型：**

- **JVM：** 考可见性、有序性、原子性，volatile、happens-before。
- **Android：** 同样重要，但要结合 Handler/Looper、线程切换、UI 主线程安全。

**集合类：**

- **JVM：** 考 HashMap 扩容、ConcurrentHashMap 分段锁/CAS。
- **Android：** 除了这些，还要结合 SparseArray（Android 特有优化）、ArrayMap 等轻量集合。

**字符串常量池：**

- **JVM：** 考 PermGen → 堆的搬家过程。
- **Android：** ART 里字符串常量池也在堆，但更关注 String 不可变性对性能和安全的影响。
